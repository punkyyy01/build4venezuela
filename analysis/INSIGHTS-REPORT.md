## 4. Per-project detail

### acopiovzla
Acopio-VZLA (also named "saman" internally) is a fully functional React SPA for registering and browsing Venezuelan earthquake aid-collection centers (centros de acopio). The app is already live on Vercel. It uses Supabase as the entire backend (Auth, PostgreSQL+PostGIS, Storage, Realtime) and implements a Twitter/Instagram-style feed of updates per center, community posts, comments, notifications via DB trigger, shelter announcements, and photo upload with client-side compression. Codebase is domain-specific, Spanish-language, and was built rapidly but deliberately over ~2 days with 23 commits. Tests exist and achieve ~99% line coverage. No CI, no server-side code beyond Supabase SQL/RLS.

- **Type:** Frontend SPA (React/Vite) backed by Supabase BaaS
- **Stack:** FE: React 18, TypeScript, Vite, Tailwind CSS, TanStack Query v5, React Router v6, shadcn/ui primitives, Lucide React | BE: Supabase (Auth, PostgREST, Realtime, Storage), PostgreSQL with PostGIS, PLpgSQL triggers and RPC functions | DB: Supabase PostgreSQL, PostGIS (geom column + GIST index for KNN), 10 versioned migrations | AI: 
- **Architecture:** Feature-sliced SPA with BaaS (no custom server) · ORM: false (Supabase JS SDK (@supabase/supabase-js) calling PostgREST and RPCs directly — no ORM. The core ML pipeline is genuinely implemented and working — this is not boilerplate. It is a FastAPI server-rendered app with SQLAlchemy/SQLite, containerized with Docker and documented for Railway deployment. The primary blocker for real deployment is the complete absence of auth: all uploaded photos (of vulnerable people) and contact details are publicly accessible to anyone with the URL, which is a serious privacy and safety concern in a real disaster scenario.

- **Type:** Web app — AI-powered family reunification via face recognition
- **Stack:** FE: Jinja2 server-rendered HTML, Tailwind CSS (CDN), Vanilla JS (search filter, loading overlay) | BE: Python 3.11, FastAPI, uvicorn | DB: SQLite (via SQLAlchemy 2.x ORM) | AI: InsightFace buffalo_l (ArcFace), ONNX Runtime (CPU), OpenCV headless, NumPy vectorized cosine similarity
- **Architecture:** Monolithic server-rendered MVC · ORM: true (SQLAlchemy 2.x DeclarativeBase ORM over SQLite; three tables: missing_persons, found_persons, matches. No migrations (uses create_all on startup).) · API: Server-rendered HTML routes via FastAPI + Jinja2; no REST/JSON API. StaticFiles mount for uploaded photos.
- **Scores:** maturity 2/5 · prod-ready 2/5 · org 4/5 · viability 3/5 · product 3/5 · diffusion 3/5
- **Signals:** 0★ · 8 commits · 985 LOC · 1 contributors · no-license
- **Real problem:** yes (critical) — Family reunification after earthquakes and displacement is one of the most urgent and emotionally devastating problems in disaster relief. The two-sided upload model — families report missing, rescuers report found — maps directly to how volunteer networks actually operate on the ground. ArcFace-based cross-matching automates what would otherwise require manual photo comparison across hundreds of reports. The problem is real, the frequency in Venezuelan disaster contexts is real, and the approach (AI-assisted face matching) is technically proven. This is not a contrived hackathon scenario.
- **Live demo:** no-public-demo — The live_url points to the GitHub repository, not a deployed web instance. No publicly accessible running application exists. A demo video is available at https://youtu.be/k4PaAmArSfU, and the project is documented for Railway deployment with Docker, so a working instance can be spun up — but none is currently public-facing.
- **Recommendation:** improve-first
- **Red flags:** No authentication anywhere — all victim photos and rescuer contact details are publicly accessible to anyone who visits the site; 0.40 cosine similarity threshold is relatively permissive; false positives in disaster contexts (wrong family match) can cause serious harm; O(N) full embedding scan on every upload — performance degrades linearly with database size; SQLite with check_same_thread=False is not safe for concurrent write load; No file size limit on photo uploads — potential for denial-of-service via large files; No deduplication — same photo uploaded twice creates a self-match; ThreadPoolExecutor limited to 2 workers — third concurrent ML request blocks; No logging — no way to audit who uploaded what or detect abuse
- **Merge potential:** Unique in the hackathon for its AI face-matching approach to family reunification. No other project is likely doing ArcFace-based cross-matching. Strong candidate to merge or build upon if auth and privacy controls are added. The facerec.py library is reusable in other contexts.

### venezuela-response-hub
A bilingual (ES/EN) static web hub that curates and links to ~20 community-built tools for the Venezuela earthquake response. Built with TanStack Start / React / Tailwind via the Lovable AI builder in under 48 hours. All content is hardcoded data — there is no database, no backend API, no auth, and no tests. The site is live at vzlaresponsehub.org and serves as an aggregator/directory, not an original tool itself. Value is entirely in curation and bilingual accessibility, not in novel engineering.

- **Type:** Static informational hub / curated directory (SSR-capable SPA, deployed as static)
- **Stack:** FE: React 19, TanStack Start, TanStack Router, shadcn/ui, Radix UI, Tailwind CSS v4, Lucide Icons, Recharts, react-hook-form, Zod | BE:  | DB:  | AI: Lovable (AI web builder used for generation)
- **Architecture:** Single-page application with optional SSR via TanStack Start/Nitro. Single route (index.tsx) contains all page logic. · ORM: false (None — no database of any kind. All data is hardcoded JS constants (SECTIONS array, EMERGENCY_NUMBERS, donation orgs, i18n dictionaries).) · API: None. No API routes. The server.ts file is only a thin SSR error-handling wrapper around TanStack Start's server entry.
- **Scores:** maturity 2/5 · prod-ready 2/5 · org 3/5 · viability 3/5 · product 3/5 · diffusion 3/5
- **Signals:** 0★ · 306 commits · 6915 LOC · 2 contributors · no-license
- **Real problem:** yes (critical) — The June 2026 Venezuela earthquake left 1,430+ dead and 68,900+ missing. Victims, diaspora, and aid workers face an acute coordination problem: dozens of community-built tools exist but are fragmented across social media and informal channels. This hub solves the discoverability and trust problem by curating 22 verified tools in one bilingual (ES/EN) directory with trust-level badges, emergency phone numbers, donation channels, and shelter finders — all organized by the user's immediate need ("I'm looking for someone", "I need shelter", "I want to donate").
- **Live demo:** working — The site loads fully at vzlaresponsehub.org. - **Merge potential:** Low. This project is a static directory/hub that links OUT to other tools rather than providing standalone functionality. Its value is in curation and bilingual UX. It could be merged as a reference implementation for a navigation layer or landing page, but it duplicates the "aggregator" role that a main hackathon portal would already serve.
